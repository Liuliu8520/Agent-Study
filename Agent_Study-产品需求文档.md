# 系统需求规格说明书

## 1. 项目概述

### 1.1 项目名称

基于大模型的个性化高数学习多智能体系统

### 1.2 项目定位

本系统面向高校学生提供个性化高等数学自适应学习服务，通过智能诊断、动态规划、RAG 微讲义生成、随堂练习判卷和智能复习形成完整学习闭环。

系统同时提供后台管理端，支持教师或管理员维护高数知识库、管理 Agent Prompt、查看学习数据分析和追踪全链路日志。

### 1.3 简历导向定位

本项目的简历定位为：

> 基于 Spring Boot 的个性化高数学习 Agent 平台，融合 RAG 知识检索、多 Agent 流程编排、数学表达式自动判卷、Prompt 热更新和学习数据分析。

项目主线面向 Java 后端开发实习，重点体现后端工程能力；AI Agent、RAG 和 Prompt 管控作为差异化亮点，可辅助投递 AI 应用开发或 Agent 工程方向岗位。

### 1.4 目标用户

学习者：

- 需要进行高数查漏补缺的大学生。
- 希望获得个性化学习计划、短讲义、练习和复习建议。

管理员：

- 教师、助教或开发者。
- 负责维护知识库、调优 Prompt、查看学习数据和排查系统异常。

### 1.5 项目目标

- 完成一个可演示、可落库、可讲解的前后端分离项目。
- 打通学生端 5 步学习闭环，而不是只做一个聊天窗口。
- 通过后台能力体现真实业务系统中的管理、审计、监控和数据分析能力。
- 通过多 Agent 编排、RAG 检索和表达式判卷体现 AI 应用工程化能力。

## 2. 需求范围与开发优先级

### 2.1 P0：必须完成的 MVP

P0 目标是保证项目能完整演示，并能作为简历项目讲清主流程。

- 学生端完成 5 步学习闭环。
- 后端真实维护 LearningState，不使用纯前端假流程。
- 接入 LLM 调用封装。
- Prompt 从数据库读取，避免硬编码。
- 使用 MySQL 持久化学习会话、讲义、练习作答、判卷结果和 Agent 调用日志。
- 使用 exp4j 完成数学表达式自动判卷。
- 完成 Vue3 前端和 Spring Boot 后端基础项目结构。

### 2.2 P1：后端工程增强

P1 目标是让项目更像完整后端工程，而不是简单 Demo。

- Spring Security + JWT 实现后台鉴权。
- Redis 存储活跃学习状态、Token 黑名单和热点 Prompt 缓存。
- 知识库切片 CRUD。
- Prompt 在线编辑、启用、版本回滚。
- Agent 调用日志、异常日志和耗时统计。
- OpenAPI / Knife4j 接口文档。
- Docker Compose 启动 MySQL、Redis 等依赖。

### 2.3 P2：AI 与数据亮点

P2 目标是增强 AI 应用和 Agent 工程方向竞争力。

- RAG 向量检索与讲义引用来源展示。
- 后台 ECharts 薄弱点 Top N 看板。
- Agent 平均响应耗时统计。
- SSE 流式输出学习过程日志。
- LLM 输出 JSON Schema 校验、失败重试和结果修复。
- 学习报告导出 Markdown。

### 2.4 P3：可选加分项

P3 不影响项目主线，时间充足时再做。

- 多模型切换。
- 学生历史学习趋势。
- 错题本。
- 题目难度自适应。
- 管理员多角色权限。
- 前端深色模式。

## 3. 学生端核心业务闭环

学生端必须严格按照线性流水线执行以下 5 个核心步骤，每一步输出进入统一 LearningState，并作为下一步输入。

```text
开始学习
  -> Step 1 智能诊断
  -> Step 2 动态规划
  -> Step 3 RAG 微讲义生成
  -> Step 4 随堂练习与自动判卷
  -> Step 5 智能复习或结业归档
```

| 步骤 | 模块 | 核心需求 | 输出 |
| :--- | :--- | :--- | :--- |
| Step 1 | 智能诊断 | 系统生成 5 道难度递进的微积分选择题，覆盖极限、导数、积分。学习者提交答案后，DiagnosticianAgent 输出薄弱知识点列表。 | weak_points |
| Step 2 | 动态规划 | PlannerAgent 根据薄弱点生成 3 天个性化学习计划，要求输出固定 JSON 结构。 | learning_plan |
| Step 3 | RAG 微讲义生成 | LessonGeneratorAgent 检索高数知识库切片，结合学习计划生成 Markdown 微讲义，内容包含定义、公式和例题。 | generated_lesson |
| Step 4 | 随堂练习 | ExerciseAgent 基于讲义生成 3 道计算/填空题，学生输入数学表达式，系统使用 exp4j 判卷。 | exercise_results |
| Step 5 | 智能复习 | ReviewerAgent 根据错误率生成复习策略。错误率 >= 50% 时生成 2 道同类型变式题；全部正确时输出结业鼓励语。 | review_result |

### 3.1 智能诊断

功能要求：

- 生成 5 道选择题。
- 题目覆盖极限、导数、积分。
- 题目难度递进。
- 学生提交答案后，系统分析薄弱知识点。
- 诊断结果必须写入 learning_session。

诊断结果示例：

```json
{
  "weakPoints": [
    {
      "code": "chain_rule",
      "name": "链式法则",
      "reason": "复合函数求导题回答错误"
    }
  ]
}
```

### 3.2 动态规划

功能要求：

- 根据 weak_points 生成 3 天学习计划。
- 每天包含学习目标、重点概念和建议练习方向。
- 输出 JSON 必须可解析、可落库、可前端展示。

### 3.3 RAG 微讲义生成

功能要求：

- 根据 weak_points 和 learning_plan 构造检索 query。
- 从知识库召回相关高数教材切片。
- 生成 Markdown 格式微讲义。
- 讲义应包含定义、公式、例题和易错提醒。
- 讲义内容需保存到 learning_session.generated_lesson。

### 3.4 随堂练习与自动判卷

功能要求：

- 基于微讲义生成 3 道计算/填空题。
- 每题包含题干、标准答案、知识点标签。
- 学生输入数学表达式作为答案。
- 系统使用 exp4j 通过随机数代入法判断学生表达式与标准答案是否近似恒等。
- 每道题的学生答案、标准答案、判卷结果和判卷详情必须落库。

判卷限制：

- 第一版只支持单变量表达式，例如 x。
- 第一版不处理复杂分段函数、隐函数、证明题和手写公式识别。
- 随机代入点需要避开表达式无定义点。

### 3.5 智能复习

功能要求：

- 统计 Step 4 的错误率。
- 错误率 >= 50% 时，生成 2 道同类型变式题。
- 全部正确时，输出结业鼓励语并归档学习记录。
- 部分错误但低于 50% 时，输出针对性复习建议并归档。

## 4. 后台管理系统需求

后台采用“AI 能力管控与数据洞察”设计理念，不只做用户增删改查。

### 4.1 后台登录与权限

功能要求：

- 管理员使用账号密码登录。
- 登录成功后返回 JWT Token。
- /api/admin/** 接口必须鉴权。
- 管理员登出后 Token 进入 Redis 黑名单。
- 默认管理员仅在开发环境自动初始化。

### 4.2 知识库管理

功能要求：

- 分页展示知识切片。
- 支持按关键词、章节搜索。
- 支持新增、编辑、删除知识切片。
- 支持启用/停用知识切片。
- 支持对单条知识切片重新向量化。
- 重新向量化后更新 embedding_json。

### 4.3 Agent 编排中心

功能要求：

- 展示 5 个 Agent 当前启用的 System Prompt。
- 支持在线编辑 Prompt。
- 支持新增 Prompt 版本。
- 支持启用指定 Prompt 版本。
- 支持回滚到历史稳定版本。
- 修改 Prompt 后无需重启后端服务即可生效。
- 所有 Prompt 修改操作必须写入 operation_log。

5 个 Agent：

- DiagnosticianAgent：智能诊断。
- PlannerAgent：动态规划。
- LessonGeneratorAgent：RAG 微讲义生成。
- ExerciseAgent：随堂练习生成。
- ReviewerAgent：智能复习。

### 4.4 数据洞察看板

功能要求：

- 聚合 learning_session.weak_points，展示薄弱知识点 Top N。
- 展示近 7 日学习会话数量。
- 展示 5 个 Agent 平均响应耗时。
- 展示 Redis 当前活跃学习会话数。
- 前端使用 ECharts 展示柱状图、折线图或统计卡片。

### 4.5 全链路日志

功能要求：

- 按学生名、日期、sessionId 查询学习会话。
- 查看某次学习会话完整 LearningState 快照。
- 查看该会话所有 Agent 调用日志。
- 查看 LLM 调用超时、JSON 解析失败、判卷异常等错误记录。
- 支持后台操作日志查询。

## 5. 非功能性需求

### 5.1 安全性

- 前台学习区免登录。
- 后台管理区必须 JWT 鉴权。
- 管理员密码使用 BCrypt 加密存储。
- 关键管理操作写入 operation_log。
- 不在代码仓库中提交真实 API Key。

### 5.2 响应速度

- 单次普通接口响应时间尽量控制在 1 秒内。
- 单次 Agent 调用目标响应时间小于 8 秒。
- 后台看板查询目标响应时间小于 1 秒。
- Agent 调用耗时必须记录到 agent_call_log。

### 5.3 并发支持

- 第一阶段目标支持 20 个学习者同时在线学习。
- LearningState 可存储在 Redis，最终结果落库到 MySQL。
- 长耗时 LLM 调用应避免阻塞关键管理接口。

### 5.4 可维护性

- 后端采用 Controller、Service、Mapper、Entity、DTO 分层。
- Agent 输入输出使用明确 DTO，不在业务层随意传 Map。
- Prompt 不硬编码在 Agent 类中。
- RAG、LLM、Embedding 客户端需要封装接口，便于后续替换。

## 6. 数据持久化要求

必须持久化的数据：

- 学习会话。
- 诊断薄弱点。
- 学习计划。
- 生成讲义。
- 练习题、学生答案、标准答案、判卷结果。
- 复习结果。
- 知识库切片与向量。
- Prompt 模板与历史版本。
- Agent 调用日志。
- 管理员操作日志。

## 7. 当前阶段暂不做

为了避免范围过大，以下内容不作为第一阶段目标：

- 不做复杂微服务拆分。
- 不做课程购买、支付、社区和消息系统。
- 不做完整 LMS 教务系统。
- 不做复杂公式编辑器，第一版使用普通文本输入数学表达式。
- 不做手写公式识别。
- 不追求生产级向量数据库，第一版使用 MySQL + Java 相似度计算即可。
- 不覆盖全部高数知识，第一版聚焦极限、导数、积分。

## 8. 第一阶段验收标准

第一阶段完成后，应满足：

- README 能说明项目介绍、技术栈、启动方式、演示账号和接口文档地址。
- 本地可启动前端、后端、MySQL、Redis。
- 学生端可完整走完 5 步学习流程。
- 后端真实调用 LLM，而不是纯前端假数据。
- 学习会话、讲义、练习作答、判卷结果和 Agent 调用日志能落库。
- 管理端能登录。
- 管理端至少支持 Prompt 查看/编辑和学习会话查询。
- 面试时能围绕 Orchestrator、RAG、Prompt 热更新、表达式判卷和日志追踪讲清楚项目难点。
